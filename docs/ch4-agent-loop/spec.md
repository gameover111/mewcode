# MewCode ch4 Agent Loop Spec

## 背景
MewCode 当前已经具备基础对话能力和工具系统。ch3 中模型可以请求工具，MewCode 执行一次工具并把结果回灌给模型，然后生成最终回复。但这个能力仍然是“一步一停”：模型拿到一次工具结果后不能继续自主观察、调整和推进任务，需要用户反复催促。

本章目标是给 MewCode 装上 Agent Loop，使它能按照 ReAct 范式自主循环：调用模型、解析响应、执行工具、回填结果、继续下一轮，直到任务完成或达到终止条件。这样 MewCode 从被动问答升级为能持续推进任务的 Coding Agent。

## 目标
- MewCode 支持 ReAct 风格 Agent Loop：一轮包含模型响应、工具执行、结果回填和下一轮判断。
- MewCode 对上层暴露统一事件流，让 TUI/CLI 能观察用户消息、模型 thinking、模型文本、工具调用、工具结果、最终回复和错误。
- MewCode 用状态机思维判断每轮后是继续还是终止。
- MewCode 支持一轮中多个工具调用，并按读类/写类分组执行。
- MewCode 支持 plan-only 模式：只允许读类工具，拦截写类工具，最终输出计划供用户审批。
- MewCode 支持外部取消和超时，循环中断时状态保持一致并返回可理解事件。
- 本章使用最小可用 system prompt，不做复杂提示词系统。

## 功能需求
- F1: 系统提供 Agent Loop，本体按 ReAct 范式执行：调用 LLM、读取响应、执行工具、回填工具结果、进入下一轮。
- F2: Agent Loop 对外输出事件流，至少包含用户消息、模型 thinking、模型文本、工具调用开始、工具结果、最终回复、错误和循环结束事件。
- F3: Agent Loop 每轮结束后根据状态判断继续或终止。
- F4: 终止条件包括模型显式结束、响应中没有工具调用、达到最大轮数上限、外部取消和不可恢复错误。
- F5: Agent Loop 支持配置最大轮数，达到上限时停止并输出可理解错误或最终状态。
- F6: Agent Loop 支持外部取消信号；取消后不再发起新的模型请求或工具执行，并输出取消事件。
- F7: Agent Loop 支持每轮或整体超时；超时后停止循环并输出超时事件。
- F8: 当模型一轮响应中包含多个工具调用时，系统能收集全部工具调用。
- F9: 系统能识别工具类别，至少区分读类工具和写类工具。
- F10: 同一轮中的读类工具可以并发执行，写类工具必须串行执行。
- F11: 工具执行结果按对应 tool call id 回填到对话历史，使模型下一轮能观察所有结果。
- F12: plan-only 模式开启时，Agent Loop 只允许读类工具执行。
- F13: plan-only 模式下遇到写类工具时，系统拦截执行，返回结构化工具结果，提示用户关闭 plan-only 后再执行。
- F14: plan-only 模式最终输出一份计划，而不是实际修改文件或执行写类操作。
- F15: 模型无工具调用时，Agent Loop 将该轮文本视为最终回复并终止。
- F16: 模型返回 thinking 内容时，Agent Loop 通过事件流透出，但不要求持久化。
- F17: 工具执行失败不会导致循环崩溃；失败结果会回填给模型，让模型下一轮决定如何调整。
- F18: Agent Loop 保留工具执行前后拦截位，供后续章节实现权限策略。

## 非功能需求
- N1: 使用 Python 实现，延续当前项目结构、测试方式和中文用户可见文本。
- N2: Agent Loop 必须保持与现有工具注册中心和 OpenAI-compatible/DeepSeek 工具调用路径兼容。
- N3: 事件流应稳定、可测试，上层不需要解析内部 Provider 事件。
- N4: 循环状态必须可观测，测试能确认每轮编号、终止原因和工具执行顺序。
- N5: 并发执行读类工具时，结果顺序必须能按原始 tool call 顺序回填，避免模型上下文错位。
- N6: 取消和超时不应留下半写入的对话状态；已经完成的工具结果可以记录，未执行的工具不能伪装成成功。
- N7: plan-only 模式必须防止写文件、改文件和执行命令这类写类/副作用工具。

## 不做的事
- 本章不做复杂系统提示词组装，只使用最小可用 system prompt。
- 本章不做完整权限策略，只保留工具执行前后拦截位。
- 本章不做 Agent 递归调用或子任务委派。
- 本章不做长期任务持久化。
- 本章不做跨进程恢复 Agent Loop 状态。
- 本章不做新的工具种类，继续使用 ch3 的六个核心工具。
- 本章不做 Claude 工具调用端到端适配，仍优先验证 OpenAI-compatible/DeepSeek 路径。

## 验收标准
- AC1: 对于需要连续两次工具观察的任务，Agent Loop 能自动执行两轮以上工具调用，无需用户再次输入。
- AC2: 当模型没有工具调用时，Agent Loop 输出最终回复并终止。
- AC3: 当达到最大轮数时，Agent Loop 停止并输出最大轮数终止事件。
- AC4: 当外部取消信号触发时，Agent Loop 停止后续模型请求和工具执行，并输出取消事件。
- AC5: 当循环超时时，Agent Loop 停止并输出超时事件。
- AC6: 事件流包含用户消息、模型文本、工具调用开始、工具结果、最终回复和结束事件。
- AC7: 模型 thinking 内容能以事件形式传给上层。
- AC8: 一轮中多个读类工具可以并发执行，并且结果按原始 tool call 顺序回填。
- AC9: 一轮中多个写类工具按顺序串行执行。
- AC10: plan-only 模式下读类工具可以执行。
- AC11: plan-only 模式下写类工具被拦截，不会修改文件或执行命令。
- AC12: plan-only 模式最终回复是一份计划，并明确需要用户审批后才能执行写操作。
- AC13: 工具失败结果会回填给模型，模型可以下一轮调整策略。
- AC14: 工具结果回填的 tool call id 与模型请求的 tool call id 一一对应。
- AC15: Agent Loop 的工具执行前后拦截位可被测试替身观察到。
- AC16: 现有 ch3 单工具任务仍能正常完成，但现在通过 Agent Loop 完成。
