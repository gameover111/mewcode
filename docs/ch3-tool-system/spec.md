# MewCode ch3 工具系统 Spec

## 背景
MewCode 当前已经具备基础终端对话能力：用户可以在 TUI 中输入问题，系统调用兼容 OpenAI 流式接口的大模型后端，并把回复流式打印出来。当前模型只能生成文本，不能读取项目、修改文件或执行命令，因此还不是能实际完成开发任务的 Coding Agent。

本章目标是给 MewCode 装上工具系统。用户提问后，模型可以通过 API 的工具调用能力表达“我要使用哪个工具和什么参数”，MewCode 负责执行对应本地工具，把结构化结果回灌给模型，再让模型基于工具结果生成回复。本章只完成一次工具调用回合：模型拿到一次工具结果后生成最终回复并停止；多轮自动 Agent Loop 留到后续章节。

## 目标
- MewCode 提供统一工具接口，所有工具都能以一致方式登记、描述、调用和返回结果。
- MewCode 支持六个核心工具：读文件、写文件、改文件、执行命令、按模式找文件、搜代码内容。
- MewCode 能把已登记工具转换成模型 API 可识别的工具列表。
- MewCode 能解析流式响应中的工具调用，包括 JSON 参数碎片拼接。
- MewCode 能执行模型请求的工具，并把工具结果作为对话历史回灌给模型。
- MewCode 在一次工具结果回灌后让模型生成最终回复，本章不做自动连环工具调用。
- 工具执行失败时，失败信息以结构化结果返回给模型，而不是让程序崩溃。

## 功能需求
- F1: 系统提供统一 Tool 接口，每个工具暴露名称、描述、参数 Schema 和执行行为。
- F2: 系统提供工具注册中心，可以集中登记工具、按名称查找工具，并输出模型 API 可识别的工具定义列表。
- F3: 系统内置读文件工具，接收文件路径，返回文件文本内容或结构化错误。
- F4: 系统内置写文件工具，接收文件路径和文本内容，写入文件后返回结构化成功结果或错误。
- F5: 系统内置改文件工具，接收文件路径、原文片段和新文本，只在原文片段唯一匹配时替换；匹配不到或匹配多次都返回清楚错误。
- F6: 系统内置执行命令工具，接收命令和可选工作目录，在超时时间内执行，并返回退出码、标准输出、标准错误和错误状态。
- F7: 系统内置按模式找文件工具，接收 glob 风格模式，返回匹配文件列表或结构化错误。
- F8: 系统内置搜代码内容工具，接收搜索文本或正则模式，返回匹配文件、行号和匹配行内容或结构化错误。
- F9: 所有文件读写、搜索和命令工作目录默认限制在当前项目工作区内；越界访问返回结构化错误。
- F10: 工具执行统一带超时、异常捕获和结构化结果，成功与失败都能被模型理解。
- F11: LLM 客户端在发起请求时携带工具定义，使模型能选择工具。
- F12: LLM 客户端能从流式响应中解析工具调用名称和 JSON 参数碎片，并在流结束后得到完整工具调用。
- F13: 当模型请求一个已登记工具时，系统执行该工具，并把工具结果回灌到对话历史。
- F14: 工具结果回灌后，系统再请求模型生成最终自然语言回复，并把回复流式打印给用户。
- F15: 如果模型没有请求工具，系统保持现有纯对话行为，直接流式打印回复。
- F16: 如果模型请求未知工具或参数 JSON 无法解析，系统把错误作为结构化工具结果回灌，并允许模型生成最终回复。

## 非功能需求
- N1: 使用 Python 实现，并延续当前项目结构和测试方式。
- N2: 新增用户可见文本、错误提示和注释使用中文。
- N3: 工具结果必须结构化，至少包含成功状态、可读摘要、数据字段和错误信息字段。
- N4: 工具执行不应泄漏 `.env`、API Key 等隐私内容；默认拒绝读取 `.env`。
- N5: 命令执行必须有默认超时，避免 TUI 长时间无响应。
- N6: 文件和搜索结果应限制输出体积，避免把过大的内容直接塞进模型上下文。
- N7: 工具系统应与具体 Provider 解耦，便于后续支持更多模型后端。
- N8: DeepSeek 作为当前真实可用模型后端，优先通过 OpenAI-compatible 工具调用格式进行端到端验证。

## 不做的事
- 本章不做多工具连续自动循环；模型拿到一次工具结果后必须进入最终回复。
- 本章不做并行工具调用。
- 本章不做用户批准每个工具调用的交互确认。
- 本章不做长期任务计划、子任务队列或后台执行。
- 本章不做 shell 会话保持；每次命令执行都是独立进程。
- 本章不做二进制文件编辑。
- 本章不做跨工作区文件操作。
- 本章不做 Anthropic Claude 工具调用端到端适配；本章以 OpenAI-compatible/DeepSeek 路径优先。

## 验收标准
- AC1: 工具注册中心能登记六个核心工具，并能按名称查找。
- AC2: 注册中心输出的工具定义包含名称、描述和 JSON Schema 参数定义，能被 OpenAI-compatible API 使用。
- AC3: 读文件工具能读取工作区内文本文件，并拒绝读取 `.env` 或工作区外路径。
- AC4: 写文件工具能在工作区内创建或覆盖文本文件，并返回成功结构化结果。
- AC5: 改文件工具在原文唯一匹配时成功替换；原文不存在或出现多次时返回清楚错误。
- AC6: 执行命令工具能返回退出码、标准输出、标准错误；命令超时时返回超时错误。
- AC7: 按模式找文件工具能返回符合 glob 模式的工作区内文件列表。
- AC8: 搜代码内容工具能返回匹配文件、行号和匹配行内容。
- AC9: 工具异常不会导致 MewCode 崩溃，而是作为结构化失败结果返回。
- AC10: 当模型流式返回工具调用参数碎片时，LLM 客户端能拼接出完整 JSON 参数。
- AC11: 当模型请求工具时，MewCode 会执行工具，把结果回灌给模型，再流式输出最终回复。
- AC12: 当模型不请求工具时，MewCode 仍保持原有纯对话行为。
- AC13: 当模型请求未知工具或给出无效 JSON 参数时，MewCode 返回结构化错误并生成最终回复。
- AC14: 端到端测试中，用户要求查看项目文件时，模型触发读文件工具，MewCode 返回工具结果并给出最终总结。
- AC15: 端到端测试中，用户要求修改文件时，模型触发改文件工具，文件内容被正确修改。
- AC16: 端到端测试中，模型最多执行一次工具调用回合，拿到结果后停止工具调用并输出最终回复。
